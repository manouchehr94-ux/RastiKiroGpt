# فلسفه توسعه پروژه Rasti Service

## گزارش روش توسعه، وضعیت فعلی و مسیر آینده پروژه

**نسخه:** 1.0
**تاریخ:** تیر ۱۴۰۵

---

# مقدمه

پروژه **Rasti Service** یک نرم‌افزار SaaS چندشرکتی (Multi-Tenant) برای مدیریت شرکت‌های خدماتی است.

هدف این پروژه صرفاً ساخت یک سایت خدماتی نیست، بلکه ایجاد یک زیرساخت پایدار و قابل توسعه برای شرکت‌هایی است که خدماتی مانند نصب، تعمیر، سرویس، بازدید و پشتیبانی ارائه می‌کنند.

از همان ابتدای پروژه تصمیم گرفته شد که کیفیت معماری، امنیت داده‌ها، قابلیت توسعه و نگهداری پروژه، بسیار مهم‌تر از سرعت توسعه باشد.

---

# هدف پروژه

هدف اصلی پروژه عبارت است از:

* مدیریت کامل چرخه سفارش
* مدیریت تکنسین‌ها
* مدیریت مشتریان
* مدیریت مالی
* مدیریت پیامک و اعلان‌ها
* سیستم چندشرکتی (Multi-Tenant)
* قابلیت توسعه برای سال‌های آینده

این پروژه قرار نیست یک نرم‌افزار تک‌شرکتی باشد، بلکه باید بتواند صدها یا هزاران شرکت را به صورت همزمان پشتیبانی کند.

---

# روش توسعه پروژه

در این پروژه روش توسعه معمول استفاده نشد.

به جای اینکه مستقیماً کدنویسی انجام شود، توسعه در چند مرحله انجام شد.

## مرحله اول

ابتدا نیاز واقعی کسب‌وکار مشخص شد.

قبل از نوشتن هر Feature ابتدا مشخص می‌شد:

* این قابلیت دقیقاً چه مشکلی را حل می‌کند؟
* کاربر چه انتظاری دارد؟
* آیا این قابلیت در آینده قابل توسعه خواهد بود؟
* آیا برای SaaS مناسب است؟

تا زمانی که پاسخ این سؤال‌ها مشخص نمی‌شد، وارد کدنویسی نمی‌شدیم.

---

## مرحله دوم

پس از مشخص شدن نیاز کسب‌وکار، معماری طراحی می‌شد.

در این مرحله موارد زیر بررسی می‌شد:

* ساختار مدل‌ها
* ارتباط بین Appها
* Tenant Isolation
* Permissionها
* امنیت
* قابلیت توسعه
* جلوگیری از ایجاد وابستگی‌های خطرناک

در این مرحله بارها تصمیم گرفته شد که Featureهای بزرگ فعلاً ساخته نشوند تا زیرساخت درست طراحی شود.

---

## مرحله سوم

پس از تأیید معماری، Prompt بسیار دقیق برای Claude Code نوشته می‌شد.

Claude وظیفه داشت:

* کد را بنویسد
* تست‌ها را ایجاد کند
* Migrationها را مدیریت کند
* خروجی را گزارش کند

اما تصمیم معماری را خودش نمی‌گرفت.

---

## مرحله چهارم

پس از پایان کار Claude، تمام خروجی‌ها مجدداً بررسی می‌شد.

در این مرحله موارد زیر کنترل می‌شد:

* کیفیت معماری
* امنیت
* قابلیت توسعه
* خوانایی کد
* وجود Code Smell
* امکان ایجاد Regression
* تأثیر روی Featureهای قبلی

اگر مشکلی وجود داشت، Feature دوباره اصلاح می‌شد.

---

## مرحله پنجم

پس از تأیید معماری، تست‌های پروژه اجرا می‌شد.

فقط زمانی یک Feature پذیرفته می‌شد که:

* تست‌ها موفق باشند.
* رفتار سیستم صحیح باشد.
* معماری نیز صحیح باشد.

---

## مرحله ششم

پس از تکمیل Feature، آن بخش Freeze می‌شد.

یعنی:

بدون دلیل مهم دیگر تغییر نمی‌کرد.

Featureهای بعدی روی همان زیرساخت ساخته می‌شدند.

---

# نقش افراد و ابزارها

در توسعه پروژه سه نقش اصلی وجود داشت.

---

## ۱- مالک پروژه

مالک پروژه مسئول:

* تحلیل نیاز کسب‌وکار
* تصمیم‌های محصول
* تعیین اولویت‌ها
* بررسی تجربه کاربری
* تصمیم نهایی

بود.

تمام تصمیم‌های محصول توسط مالک پروژه گرفته شد.

---

## ۲- Claude Code

Claude مسئول انجام عملیات برنامه‌نویسی بود.

وظایف Claude:

* تولید کد
* اصلاح کد
* ساخت تست
* اجرای تست
* ساخت Migration
* گزارش تغییرات

Claude مانند یک برنامه‌نویس ارشد عمل می‌کرد.

---

## ۳- ChatGPT

ChatGPT نقش برنامه‌نویس صرف نداشت.

نقش ChatGPT شامل موارد زیر بود:

* Principal Architect
* Technical Lead
* QA Architect
* Technical Project Manager

وظایف اصلی:

* بررسی معماری
* تحلیل ریسک
* جلوگیری از ایجاد Technical Debt
* طراحی Featureها
* بازبینی خروجی Claude
* پیشنهاد Refactor
* طراحی ساختار توسعه
* طراحی مسیر آینده پروژه
* بررسی امنیت و Multi-Tenant
* تصمیم‌گیری درباره Freeze شدن Featureها

در بسیاری از موارد، ChatGPT پیشنهاد Claude را رد کرد و راه‌حل استانداردتری انتخاب شد تا معماری پروژه در بلندمدت دچار مشکل نشود.

# مسیر آینده پروژه

پس از پایان این مرحله، تمرکز توسعه روی موارد زیر خواهد بود:

1. بازبینی کامل چرخه سفارش (Order Lifecycle)
2. بررسی کامل Permissionها و Roleها
3. تکمیل فرآیندهای مالی باقی‌مانده
4. توسعه Customer Portal
5. افزودن Push Notification واقعی
6. توسعه نسخه موبایل/PWA
7. بهینه‌سازی عملکرد و مانیتورینگ
8. مستندسازی کامل معماری هر Feature

# نتیجه‌گیری

پروژه Rasti Service با رویکرد «معماری‌محور» توسعه یافته است، نه «کدنویسی‌محور».

در این روش، ابتدا نیاز کسب‌وکار و معماری بررسی می‌شود، سپس پیاده‌سازی انجام می‌شود، بعد از آن تست، بازبینی، اصلاح و در نهایت Freeze صورت می‌گیرد.

هدف این رویکرد، تولید نرم‌افزاری است که سال‌ها قابل نگهداری، توسعه و استفاده باشد، بدون اینکه با رشد پروژه دچار آشفتگی معماری یا افزایش بدهی فنی شود.
